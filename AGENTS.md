# AGENTS.md

Этот файл помогает AI-агенту быстро понять проект и не делать лишних, опасных или неуместных изменений.

## Что обычно прописывают в AGENTS.md

Обычно в таком файле фиксируют:
- что это за проект и какой у него основной сценарий;
- какой стек используется;
- как запускать проект локально;
- где лежат ключевые модули и шаблоны;
- какие продуктовые и UI-договоренности уже приняты;
- что агенту можно и нельзя менять без отдельного запроса;
- как проверять результат после изменений;
- какие стилистические правила важны для кода, текстов и интерфейса.

## Контекст этого проекта

Проект: `Pigly`

Это отдельный Django-проект под X/Twitter workflow.

Основной сценарий:
- пользователь открывает лендинг;
- устанавливает браузерное расширение;
- регистрируется или входит через сайт;
- в будущем использует действия расширения прямо в интерфейсе X/Twitter;
- ключевые функции продукта: сокращение текста поста и генерация AI reply.

## Стек

- Python
- Django
- SQLite локально
- `python-decouple`
- Django templates
- Alpine.js
- HTMX
- Tailwind через CDN

## Ключевые части проекта

- `config/settings.py` — настройки проекта
- `config/urls.py` — корневая маршрутизация
- `core` — лендинг, базовые страницы, shared UI
- `users` — регистрация, вход, профиль, пользовательская модель

## Поведение агента в этом репозитории

Агенту стоит:
- сначала читать существующую реализацию, а не переписывать все с нуля;
- сохранять текущую архитектуру Django-проекта;
- вносить минимально достаточные изменения;
- не переименовывать файлы, маршруты и публичные сущности без необходимости;
- не удалять чужие изменения без явного запроса;
- не тащить в проект лишние зависимости без причины;
- учитывать, что проект сейчас на ранней стадии и важнее скорость и ясность, чем сложная архитектура.

## Важные продуктовые договоренности

Сейчас продукт строится вокруг первого экрана:
- слева — установка расширения;
- справа — регистрация или вход;
- на первом экране без скролла должны быть видны только установка расширения и auth-блок;
- верх страницы должен быть минималистичным: логотип слева, язык и тема справа;
- не перегружать hero лишними элементами;
- любые декоративные изображения добавлять точечно, а не во все карточки подряд.

## UI-договоренности

- интерфейс должен выглядеть чисто, минималистично и спокойно;
- не превращать лендинг в перегруженный маркетинговый экран;
- первая секция должна быть самой важной визуально;
- блоки ниже первого экрана должны быть вторичными по акценту;
- не ломать мобильную версию;
- не делать слишком много одинаковых белых карточек подряд без визуального ритма;
- темная тема должна работать аккуратно, без контрастной грязи.

## Правила для текстов и кодировки

- использовать UTF-8;
- не допускать битой кириллицы;
- не смешивать русский и английский без причины;
- если меняются пользовательские тексты, проверять итоговый рендер на странице;
- учитывать, что часть текстов на лендинге подставляется через JS-словарь `RU/EN`.

## Правила для изменений кода

- предпочитать точечные правки;
- не делать большой рефакторинг без запроса;
- не менять auth-flow без отдельного согласования;
- не добавлять backend-логику "на вырост", если она пока не нужна;
- перед изменениями лендинга учитывать desktop и mobile одновременно;
- если добавляются изображения, смотреть, не ломают ли они иерархию и читаемость.

## Рекомендуемые агенты для ускорения работы

Если в проекте используется несколько агентов параллельно, на текущем этапе лучше делить работу так:

### 1. Landing/UI агент

Зона ответственности:
- `templates/core/landing.html`
- `templates/base.html`
- визуальная иерархия первого экрана;
- desktop/mobile верстка;
- темы и визуальные состояния.

Что делает:
- правит композицию лендинга;
- настраивает отступы, карточки, акценты, плашки, CTA;
- следит, чтобы первый экран без скролла показывал только установку расширения и auth.

Что не делает:
- не меняет backend-логику;
- не трогает auth-flow;
- не добавляет серверную бизнес-логику.

### 2. Backend/Auth агент

Зона ответственности:
- `users`
- `config/settings.py`
- `config/urls.py`
- модели пользователя, профиль, auth-маршруты, сессии, связанная backend-логика.

Что делает:
- развивает регистрацию и вход;
- подготавливает логику связки сайта с расширением;
- пишет backend для пользовательских настроек, профиля, лимитов, access-control;
- в будущем готовит endpoint'ы для extension session/token.

Что не делает:
- не ломает текущую структуру лендинга;
- не меняет UI без необходимости;
- не трогает стили и декоративную часть, если это не нужно для формы.

### 3. AI/API агент

Зона ответственности:
- `assistant`
- будущие API для `shorten` и `AI reply`
- контракт между сайтом, расширением и backend.

Что делает:
- проектирует и реализует endpoint'ы для генерации;
- задает формат запросов и ответов;
- подготавливает integration layer под LLM;
- следит, чтобы API было простым и пригодным для extension.

Что не делает:
- не переписывает auth;
- не вмешивается в layout лендинга;
- не занимается маркетинговым контентом.

### 4. History/Profile агент

Зона ответственности:
- `history`
- профиль пользователя;
- пользовательские предпочтения;
- история генераций.

Что делает:
- строит модели истории;
- добавляет просмотр истории и базовых настроек;
- готовит основу под сохранение тона, языка и пользовательских действий.

Что не делает:
- не трогает первый экран лендинга;
- не меняет API генерации без согласования с AI/API агентом.

### 5. QA/Polish агент

Зона ответственности:
- регрессионная проверка;
- тексты;
- кириллица;
- визуальные и UX-мелочи.

Что делает:
- проверяет desktop/mobile;
- отслеживает поломанную кириллицу;
- проверяет `manage.py check`;
- смотрит, не поехали ли формы, переключатели, тема, `RU/EN`;
- помогает быстро находить слабые места после правок других агентов.

Что не делает:
- не переписывает большие куски логики;
- не делает архитектурные решения за другие роли.

## Как лучше делить работу между агентами

На текущем этапе самый полезный набор параллельной работы такой:
- агент 1: `Landing/UI`
- агент 2: `Backend/Auth`
- агент 3: `AI/API`

Если нужно усилить продуктовую часть без хаоса:
- отдельно подключать `History/Profile`
- в конце прогонять `QA/Polish`

## Что сейчас важнее всего

Если выбирать приоритет именно сейчас, то порядок такой:
1. `Landing/UI` — довести первый экран до чистого и сильного состояния
2. `Backend/Auth` — закрепить нормальный auth-flow и будущую связку с расширением
3. `AI/API` — подготовить backend под `shorten` и `AI reply`

## Что проверять после изменений

Минимум:
- проект запускается;
- `manage.py check` проходит без ошибок;
- главная страница открывается без traceback;
- не ломается верстка первого экрана;
- не поехала кириллица;
- светлая и темная тема не разваливают layout;
- переключение `RU/EN` продолжает работать.

Если менялся UI лендинга, дополнительно проверить:
- desktop first screen;
- mobile first screen;
- кнопки установки расширения;
- переключение `Регистрация / Вход`.

## Предпочтительный стиль работы

- сначала понять контекст;
- потом внести минимальное рабочее решение;
- после правок кратко объяснить, что изменено;
- если правка спорная визуально, опираться на уже согласованную композицию страницы, а не изобретать новый стиль с нуля.
